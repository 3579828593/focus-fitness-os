PRAR 四阶段详解

Perceive 感知

收集当前项目状态、识别待解决问题、分析运行环境。包括读取代码库、检查部署状态、识别错误日志等。

Reason 推理

基于感知信息进行推理分析，设计技术方案。权衡不同方案的利弊，制定实施计划，预见潜在风险。

Act 行动

执行技术方案：编写代码、修改配置、部署服务、运行测试。每个行动都有明确的验证标准。

Reflect 反思

总结经验教训，将可复用的知识提炼为 Skill。记录踩过的坑，更新技能注册表，为下一周期积累认知资产。

核心设计理念
PRAR 循环不是简单的试错迭代，而是认知资产的积累过程。每个周期的 Reflect 阶段将隐性知识转化为显性技能（Skill），使得后续周期可以直接复用前序经验，实现认知复利。

跨平台策略
Single Codebase - Multiple Platforms

采用单一代码库多平台输出策略，通过 Conditional Imports 实现平台差异化适配，90%+ 代码在 Android 和 Web 之间共享。

90%+

代码复用率

共享业务逻辑、UI 组件、状态管理，仅平台适配层有差异代码。

2 Platforms

Android APK（移动端原生）+ Web（浏览器 SPA，部署于 GitHub Pages）。

Conditional Imports

适配机制

Dart Conditional Imports 编译时选择平台实现，零运行时开销。

TTS 抽象层

Text-to-Speech 功能通过抽象接口实现跨平台适配：

// 平台无关接口
abstract class TtsService {
  Future<void> speak(String text);
  Future<void> stop();
}

// Native 实现 (flutter_tts)
class FlutterTtsService implements TtsService { ... }

// Web 实现 (dart:html SpeechSynthesis)
class WebTtsService implements TtsService {
  // window.speechSynthesis (nullable in Dart 3.12)
}

Database 抽象层

Drift 数据库通过不同的 Connection 实现实现跨平台：

// Native: NativeDatabase (sqflite)
LazyDatabase(() async {
  final dbFolder = await getApplicationDocumentsDirectory();
  final file = File(p.join(dbFolder.path, 'app.db'));
  return NativeDatabase(file);
})

// Web: WebDatabase (drift/web.dart)
LazyDatabase(() async {
  return WebDatabase('app_db');
})
重要发现
drift_web 不是独立包。Drift 的 Web 支持内置于主 drift 包中，通过 package:drift/web.dart 导入。drift_web 包不存在，添加到 pubspec.yaml 会导致解析失败。

关键教训
Lessons Learned - Pitfalls & Solutions

以下是 5 个 PRAR 循环中积累的关键技术教训，每一条都来自真实的调试实践。

CriticalDrift

drift_web 不是独立包

Drift 的 Web 支持内置于主 drift 包中，通过 package:drift/web.dart 导入。在 pubspec.yaml 中添加 drift_web 依赖会导致解析失败。

Breaking ChangeFlutter 3.29+

--web-renderer flag 已移除

Flutter 3.29+ 移除了 --web-renderer 命令行参数。CanvasKit 现在是默认且唯一的 Web 渲染器，无需也无法再手动指定。

dart2jsDart 3.12

@staticInterop 兼容性问题

dart:js_interop 的 @staticInterop 模式在 Dart 3.12 + dart2js 组合下可能编译失败。解决方案：改用 dart:html 直接调用浏览器 API。

Nullabledart:html

SpeechSynthesis 可空类型

在 Dart 3.12 中，window.speechSynthesis 返回 SpeechSynthesis?（可空类型），必须进行 null 检查后才能调用方法。

dart2jsConditional Imports

私有函数跨平台解析失败

Conditional imports 中使用 _ 前缀的私有函数在 dart2js 编译时可能无法正确解析。解决方案：使用 public 函数名（去掉 _ 前缀）。

YAMLGitHub Actions

YAML 禁止中文注释

GitHub Actions 的 YAML 文件中使用中文注释会导致解析失败（encoding error）。所有注释必须使用英文。这是一个反复出现的问题。

Node-REDflows.json

wires 格式必须为数组嵌套

Node-RED flows.json 中 wires 字段格式必须为 [["node_id"]]（数组的数组），即使只有一个连接目标。格式错误会导致节点连线丢失。

DockerENTRYPOINT

ENTRYPOINT 会清除 base 入口

Docker 中使用 ENTRYPOINT [] 空数组形式会清除 base image 的 entrypoint.sh。应使用 ENTRYPOINT ["command", "arg"] 完整形式，或使用 CMD。

SOLO-OS DAG 规范
DAG Declaration - Soft Routing vs Hard Scheduling

SOLO-OS 允许将 PRAR 循环声明为 DAG（有向无环图），实现工作流的结构化编排。DAG 声明支持两种执行模式：软路由与硬调度。

两种执行模式

Soft Routing

软路由 (Agent reads spec)

Agent 读取 DAG 规范作为指导，自行决定执行顺序和策略。灵活度高，适合探索性任务和需要人类判断的场景。DAG 作为"路线图"而非"硬约束"。

Hard Scheduling

硬调度 (Compiler executes DAG)

编译器/调度器严格按 DAG 拓扑顺序执行节点，每个节点完成后自动触发后继节点。确定性高，适合 CI/CD 等需要可重复执行的场景。
